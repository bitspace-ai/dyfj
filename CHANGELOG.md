# Changelog

Notable changes to DYFJ. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

DYFJ is an actively developed prototype with no release tags yet, so entries are dated rather than versioned. Document-level revisions of the operating-context README are tracked separately in its Revision history section.

## [Unreleased]

### Added

- Hosted multi-model daily-driver ergonomics: a standing **paid-inference posture** (`approvePaidDefault` / `[paid].approve_paid_default` / `DYFJ_APPROVE_PAID_DEFAULT`, default off) and a **budget-ceiling warn-then-confirm** path. When a loopback turn omits `approvePaidInference`, the engine falls back to the configured default; explicit per-turn opt-in/out (`--approve-paid`) still wins, and non-loopback transports never inherit the standing posture. Budget session/per-call ceilings (`defaultSessionBudgetUsd` / `defaultPerCallBudgetUsd`) gain a config-file layer (`[budget]` in `~/.dyfj/config.toml`) alongside the existing env bindings; when projected spend would cross a ceiling the runtime warns with the numbers and requires explicit confirmation before proceeding (reusing the UDS mid-turn `approval` round-trip — the CLI prompts on stderr). Without an interactive approver it fails closed, matching the existing approval-gate posture. The HTTP and UDS servers now load engine config at boot and thread it through the shared turn core.
- In-session model switching in the `dyfj` REPL: `/model` prints the active model and available slugs; `/model <slug>` validates against `models/list` and mutates the active routing for subsequent turns without relaunching or changing the paid posture.

### Changed

- Compiled `dyfj` CLI net grants for Deno 2.9 dynamic UDS paths. Deno 2.9 requires an exact-match `unix:` net grant per socket path; the compiled binary still bakes the default `~/.dyfj/run/workbench.sock` grant at compile time for speed, but `deno task compile-cli` now ships a thin launcher (`dist/dyfj`) that execs the compiled `dist/dyfj-bin` on the default path and falls back to `deno run` with a runtime-resolved `unix:<sock>` grant when `DYFJ_SOCKET` or `XDG_RUNTIME_DIR` shifts the path (mirroring the serve-unix bind-side pattern). Net permission stays scoped to loopback HTTP plus the one UDS path in use; no literal host path is committed. Regression coverage lives in `scripts/dyfj-launcher.test.ts`.
- Budget enforcement at the spend boundary is warn-then-confirm rather than an immediate hard deny when an approver is available; non-interactive transports still fail closed at the ceiling.
- Budget-ceiling confirmation is deduped to once per turn for the same projected spend level (preflight + per-call gate no longer double-prompt); a later agent-loop call re-prompts only when projected spend rises above what was already confirmed.
- Budget-ceiling dedupe tracks per-call and session projections independently so a later call that crosses the session limit re-prompts even when per-call was already confirmed at the same estimate.

- A declared configuration key registry (`CONFIG_SCHEMA` in `prototype/src/config.ts`) — the single source of truth for the system's env surface. Each key declares its env-var binding, its **engine vs client domain** (the engine owns runtime/data/secret config; the engine-free CLI owns its transport/routing slice), its type, and whether it is a **secret pointer** (a key carrying a credential — an `op://` ref / keychain item / env-var name — resolved at point of use, never stored as a plaintext value on the config surface). `DYFJ_PRINCIPAL_ID` is declared but flagged `sessionState`: per the configuration-system thesis the principal rides the connection (the UDS peer's OS identity locally, the tailnet identity remotely), so it is *not* config — it is declared only so the permission allowlist can account for it. Two consumers move onto the declared surface in this slice (below); migrating the remaining env-read sites continues incrementally.
- The permission-parity check became **structural**. The hand-maintained `deno.json` permission `env` arrays are now asserted against the declared `CONFIG_SCHEMA` surface (forward: every declared engine env var is granted by each turn-running engine profile — `workbench`, `workbench-http`, `serve-unix` — so no profile can silently lag the runtime, the live failure that surfaced `DYFJ_PRINCIPAL_ID` over the socket; reverse: no profile may grant an undeclared runtime env var, so a new var can't reach the allowlist without joining the declared surface). This replaces the band-aid `serve-unix ⊇ workbench-http` env parity test (moved from `uds-server.test.ts` to `config.test.ts`); the `serve-unix ⊇ workbench-http` **net** backstop is retained until net-host derivation from the schema lands.
- Budget defaults migrated onto the declared config surface. `budget.ts` no longer inlines the `DYFJ_BUDGET_SESSION_USD` / `DYFJ_BUDGET_PER_CALL_USD` names and the `$1.00` / `$0.10` literals — those live in `CONFIG_SCHEMA` and resolve through `resolveBudgetDefaultsFromEnv` (defaults → env, rejecting non-numeric or negative limits rather than silently coercing to `NaN`). They are resolved **once at the runtime boundary** (`resolveRuntimeEnvDefaults`, the same place the principal, root, and budget-tally mode resolve) and threaded into the turn, so the core reads no env for budget; precedence is per-turn override → boundary-resolved default → the declared `BUDGET_DEFAULTS`. `DYFJ_PRINCIPAL_ID` resolution is likewise centralized behind one `resolvePrincipalId` resolver (the boundary plus the deep event-attribution fallback), retiring the scattered `process.env.DYFJ_PRINCIPAL_ID ?? process.env.USER` reads. The budget config-*file* layer and wiring `loadConfig` into the CLI/HTTP entrypoints (today only `serve-unix` loads the file) are the next config slice.

- An `operator` permission profile that auto-approves contained mutating tools instead of prompting per-call (`permissionLevel` config, default `strict`). The mid-turn approval gate is fail-closed but unusable for daily companion use — every `write_file` (and soon `edit_file`/`bash`) prompts. The config knob now drives the command policy: on a **loopback/operator turn** with `permissionLevel = "operator"`, a *contained* mutation (local, free, workspace-write) is auto-approved with its own audit basis (`policy:allow:operator-profile`); paid, networked, or command-execution tools are deliberately **not** covered and still prompt, so a future `bash` stays gated until its own safety model says otherwise. `strict` (the default) keeps the per-call approval gate exactly as-is. The level resolves once at the boundary and threads to `evaluateCommandPolicy` alongside the loopback transport — a non-loopback consumer never receives the relaxed posture. Verified end-to-end: under `operator` a non-interactive write is auto-approved and written; under `strict` the same write is denied.

- Typed engine configuration with a TOML config file (`~/.dyfj/config.toml`). The runtime's startup posture — the defaults it uses when a request doesn't specify — now resolves from a single declared surface with precedence **defaults → config file → environment**, replacing scattered per-consumer env reads for the first knobs. The file is TOML: hand-edited, comments, idiomatic for the future Rust core. Config holds startup posture, never session state; anything the app writes (machine state) lives elsewhere, so the hand-edited file stays pristine and comment-preserving writes are a non-problem. First two knobs: `defaultCompanionModel` (the model used on a "bare turn" with no model/tier/hint — set a frontier slug for the companion; threaded through `selectWorkbenchModel` ahead of the registry's local default) and `permissionLevel` (`strict` | `operator`, default `strict`; the policy honoring `operator` lands with the permission-profile work). Loaded **once at the engine boundary** (the UDS server) and passed into the runtime — no hidden per-turn reads; a malformed config fails the boot loudly rather than silently mis-configuring. The TOML parser is imported lazily so the module loads cleanly under the test runner. Migrating the remaining env-read sites and deriving the `deno.json` permission allowlist from the schema are follow-on slices.

- `search_memory` — vendor-neutral recall over an external long-term memory. The turn path can now query an operator-configured external memory by meaning (semantic search) on demand, complementing the curated context it injects. DYFJ owns the capability (the `search_memory` tool); the backend is a config value, never a vendor named in the code — any memory that speaks MCP and exposes a search tool is reachable by setting `DYFJ_MEMORY_MCP_URL` (with `DYFJ_MEMORY_MCP_TOOL`, default `search`, and an optional `DYFJ_MEMORY_MCP_TOKEN`), so swapping the backing store is a config change (or, for a non-MCP backend, one adapter behind `MemorySearch`), never a rename through the codebase. Protocol-as-firewall applied to memory: bet on the protocol, keep vendor optionality. The tool is read-only, opt-in (an unset URL means it is never offered), and offered only on a loopback/operator turn — a non-loopback consumer never receives it, so the external memory is unreachable off-box. Its egress gets a dedicated `recall` policy tier (`policy:allow:operator-configured-recall`): auto-allowed without a per-call approval prompt, because the destination is operator-configured and fixed (the model chooses the query, not where it goes) and the tool is registered only for permitted turns. The MCP SDK is imported lazily so the capability adds no load-time cost and the module loads cleanly under the test runner. Enabling recall requires granting the runtime network access to your endpoint.

- Memory injection is now scoped by a curated **inject classification**, not by type (`schema/024_memories_inject.sql`). The turn-mode path previously bulk-injected the full content of every `user`+`feedback` memory on every turn (~63KB / tens of thousands of tokens), filtered only by privacy clearance — with no relevance or recency scoping. That corpus was a frozen one-time seed with no live source, so it injected stale and retired material (defunct vocabulary, ephemeral dated session-state checkpoints) as if current: staleness as a correctness failure, not just token cost. The `memories` table gains an `inject` column (`always` | `index` | `never`, default `index`): `always` is the small curated worldview (identity core + operating preferences) injected as content; `index` is slug+name+description only, pulled on demand via `read_memory`; `never` is withheld from both. The runtime loads `inject='always'` as the core context and `inject='index'` as the pull-on-demand index (`loadInjectedMemories` / `loadIndexedMemories`), each still clearance-scoped — replacing the former type-based bulk load. Injection drops from the full personal pile to a tight, deliberately-curated set; everything else stays reachable without being dumped. The durable source fix — regenerating the curated rows from a maintained context source of truth instead of a frozen seed — is the next slice.
- The `dyfj` CLI can approve mutating tools over the `--unix` seam. When a turn over the socket hits a tool that needs approval (e.g. `write_file`), the runtime's mid-turn `approval` request is answered by an interactive `y/N` prompt that shows the tool, its arguments, and a content preview — printed to stderr so a `--json` turn's stdout stays clean. Non-interactive runs (no TTY) deny without prompting, fail-closed. `connectUnixClient` gains an `onApproval` hook that registers the peer's `approval` request handler; with no approver the server's request fails and it denies. This makes an approved `write_file` over `dyfj --unix` a live, human-in-the-loop operation.
- Mutating tools behind a mid-turn approval gate — `write_file` + the approval round-trip. The command policy already routes write-filesystem to `ask`; the runtime now resolves `ask` by calling an injected `confirmToolApproval(request) → approve | deny`, executing only on approval (authz `policy:allow:operator-approved`) and otherwise denying (`policy:deny:approval-denied`). The default is **deny** — fail-closed, like the paid-escalation gate — so a missing approver never runs a mutation, and a malformed call is rejected before approval is even requested. Over the UDS/JSON-RPC seam the approver is a server-initiated `ctx.request("approval", {tool, callId, arguments, title})` answered by the connected client (the mid-turn duplex round-trip the seam was built for); HTTP/SSE has no such channel, so mutating tools deny there. The first mutating tool, `write_file`, writes UTF-8 within the workspace root (symlink-safe; parent must exist), registered alongside the read-only file tools. Payload-bearing arguments are redacted from the persisted tool-call event and session replay — a `redact` schema flag replaces the value with a constant `[redacted]` sentinel (any runtime type, including malformed or denied calls), so `write_file` content never reaches the durable log or replay (CWE-532) — centralized so future mutating tools inherit it. `bash` + `edit_file` ride this gate in a later slice.
- Transport seam — interactive `turn` over the JSON-RPC/UDS duplex channel. The security-sensitive turn path (request resolution, per-session serialization, resume reconstruction, paid-escalation gating, runtime invocation) was lifted out of the HTTP driver into a transport-neutral `turn-runner.ts`, so the HTTP/SSE veneer and the UDS/JSON-RPC seam run the identical turn through one shared core instead of duplicating the money/audit/clearance logic per transport — a pure refactor proven faithful by the existing HTTP suite (the transport-parity and per-session-lock TOCTOU tests stayed green). The UDS server gains the `turn` method on that core: intermediate text deltas and runtime events stream back as `stream` notifications whose params reuse the HTTP SSE frame shape, so a client shares one frame handler across transports, and the receipt is the RPC result. UDS is the canonical `loopback` transport, so paid inference and per-turn budget overrides remain available — but only with the explicit per-turn opt-in the shared core enforces (loopback + opt-in to spend; denied otherwise) — covered by tests for paid approve/deny, budget override, streaming, and invalid params. RPC handlers also gained a per-connection context (`notify` + a server-initiated `request`) so a running handler can both stream and issue the mid-turn approval request — the mechanism the approval keystone builds on.
- Transport seam — JSON-RPC 2.0 core and Unix-domain-socket transport. A hand-rolled, SDK-free JSON-RPC 2.0 core (`jsonrpc.ts`: newline-framed wire, message classification, dispatch, and the DYFJ fail-closed application error codes) and a bidirectional peer (`jsonrpc-peer.ts`) over any byte-stream connection, with a UDS server (`uds-server.ts`) serving the read methods (`models/list`, `sessions/list`, `events/query`) through the same runtime functions the REST endpoints use. The engine-free `dyfj` CLI reaches those methods over the socket (`--socket` / `DYFJ_SOCKET`), and a `serve-unix` launcher (plus root-level `serve-unix` / `cli` task proxies) serves the seam; the socket path resolves `DYFJ_SOCKET` → `$XDG_RUNTIME_DIR/dyfj` → `~/.dyfj/run`, with the parent directory created mode 0700 and stale-socket clearing that refuses to unlink a non-socket. UDS is the canonical `loopback` transport per the transport-seam decision (2026-06-21); a committed runnable spike (`examples/uds-jsonrpc-spike.ts`) proves the duplex round-trip.
- Live streaming through the agent loop + a thrash guard. The OpenAI-compatible streaming reader now parses tool calls out of the SSE stream (indexed `tool_call` deltas, accumulating fragmented arguments), so a streamed turn can both stream text and request tools. The runtime streams tool-offering calls on providers whose wire path supports it (`modelStreamsToolCalls` — local MLX/Ollama + hosted OpenAI; Anthropic/Google stay buffered, since their streaming readers don't carry tool calls, and tool calls are captured from buffered JSON there). This also restores live token streaming for ordinary companion replies — which the Slice A file-tool registration had silently disabled by suppressing streaming whenever tools are projected — and streams each gather step live. Adds an anti-thrash guard: when the model emits a whole step of tool calls it already made this turn, the loop drops tools and forces a concluding answer instead of spending steps up to the cap. Verified live on the local 30B coder: streamed tool calls are captured and executed, normal replies stream, and a multi-hop task that previously ran to the 8-step cap now concludes in three.
- Agent loop — multi-step tool iteration. The runtime now loops model↔tools until the model stops requesting tools or a step cap (`MAX_TOOL_STEPS`, 8) is reached, replacing the single tool round (`src/workbench.ts`). Tool-gathering steps run buffered (the streaming readers don't parse tool-call deltas yet — tracked as a follow-up), with progress surfaced via a per-step log and the `tool_call`/result events; at the cap the runtime drops tools to force a concluding answer, which streams. The follow-up prompt accumulates results across steps and tells the model not to repeat calls it already made. Verified live against the local Qwen3-Coder-30B: a batched single step, genuine sequential multi-step iteration, and the cap's forced conclusion. Regression-tested with a mock provider (natural termination + cap). Budget enforcement and accounting are per provider call, not per turn: every call in the loop is budget-checked before and recorded after, so per-call and session limits hold across steps and receipts/audit events aggregate every call — closing a pre-push finding that the loop could make several hosted calls under one preflight and record only the final one's usage.
- Agent loop Slice A — read-only workspace file tools. The runtime offers `read_file` and `list_files` (`src/file-tools.ts`), path-scoped to the workspace root (`DYFJ_ROOT`, else cwd) with traversal/escape rejected, so the companion can read and reason over the project it's working in. They're read-only and side-effect-free, so `evaluateCommandPolicy` now auto-allows read-only filesystem access (`filesystem: "read"`); write filesystem still falls through to `ask`/`deny` until the Slice B safety model and Slice C approval flow land. Registered in turn mode alongside `memory.read`. Verified end-to-end: the companion calls `read_file` and answers from the file's content.
- `dyfj` CLI gains `--mode <turn|ask|next-work>` plus a `dyfj ask "<prompt>"` shorthand, so the client can request repo/operator context (ask mode) instead of only the companion turn path. Ask mode pulls the lightweight repo context (AGENTS/README/notes/Beads) and populates `context.sources`, fixing the case where "summarize this repository" over the default turn path returned a summary of the personal corpus instead. Default remains `turn`.
- Privacy-class scoping for memory injection (`schema/019_memories_visibility.sql`): the `memories` table gains a `visibility` column (`private` | `shareable` | `client_safe` | `public`, default `private`, per the AGENTS.md taxonomy). The turn-mode companion path now derives a clearance from the request's auth-context transport (`memoryClearanceFor`) and filters injected memory by it — a loopback/in-process operator sees the full corpus; any non-loopback consumer (remote or shared, even with the bearer, since a shared key does not prove identity) is limited to `client_safe` + `public`. Existing rows default to `private`, so the personal corpus cannot reach a non-loopback surface until a memory is deliberately reclassified. Closes the leak a loopback CLI test surfaced (an "operator summary of the repository" returned a summary of the operator's private corpus). The read-memory tool is transitively gated via the (now-filtered) `allowedMemorySlugs`. Per-principal clearance (a tailnet-identified operator → full corpus) is deferred to the Tailscale-identity work.
- System prompts persist in a dedicated Dolt `prompts` table (`schema/017_prompts.sql`) — authored, versioned, trusted config kept separate from the untrusted memory layer (the trust anchor must not share a store with injectable prompt material). A `prompts.ts` loader reads the active companion base prompt with a hardcoded fallback; `buildAskSystemPrompt` now composes that loaded persona with live context instead of hardcoding it. The default companion prompt is positively framed (capable collaborator, context available not binding), which fixes the prior repo-scoped prompt that made the default mode refuse general work (e.g. "I can't write Rust… outside repo scope").
- Google Generative AI (Gemini) provider adapter: native `generateContent` / `streamGenerateContent` wire format with `x-goog-api-key` auth, `systemInstruction`/`contents`/`generationConfig` requests, and `candidates`/`usageMetadata` responses; behind the paid-escalation gate, fails closed without `GEMINI_API_KEY`. Current Gemini rows added (`schema/015_models_gemini_2026_06.sql`): `gemini-3.5-flash` (tier 1) and `gemini-3.1-pro` (tier 2); the stale `gemini-2.5-*` rows stay deactivated. Tool calling is deferred (Gemini turns are text/JSON only for now).
- Hosted OpenAI inference lane: the `openai-completions` adapter now serves a hosted path (https base URL + `OPENAI_API_KEY` bearer) alongside the local path, via a shared `executeOpenAICompatibleTurn`. GPT rows added to the registry (`schema/014_models_openai_2026_06.sql`): `gpt-5.4-mini` (tier 1), `gpt-5.4` and `gpt-5.5` (tier 2). Hosted GPT inherits the existing paid-escalation gate; tier>0 fails closed over HTTP.
- `deno task check`: strict typecheck of the non-test import graph (entrypoints `src/http.ts`, `src/workbench.ts`, `mcp/server.ts`). `deno task test` now runs it first, closing the gap where vitest's esbuild transpilation let type errors accumulate unenforced.
- Session REST surface for the cockpit: `GET /api/sessions` (grouped by project), `POST /api/sessions` (create bound to a project), and `GET /api/sessions/{id}/events` with an optional `asOf` Dolt time-travel read. Sessions gain a `project` column (`schema/013_sessions_project.sql`).
- Multi-turn conversations: `POST /api/turn` accepts a `sessionId` to resume — events append to the session and a compact transcript is rebuilt from prior `session_start`/`model_response` events so the model carries the conversation. The operator prompt now rides on `session_start` events to make transcripts reconstructable from Dolt alone.

### Removed

- The unused `settings.example.json` config stub and its setup-doc references. The README told operators to `cp settings.example.json settings.json`, but nothing in the code ever read it, and its contents were stale (it named the retired `Qwen3.5-4B-8bit`). Configuration is entirely environment-driven today (`.env` + the Deno task permission profiles); a real configuration system is tracked separately as follow-up work.
- Vestigial schema and its MCP tools (`schema/018_drop_vestigial.sql`): the `reflections` and `skills` tables and the unused capability/discovery event scaffolding from `010` (4 enum values, 5 nullable columns, 2 indexes on `events`) are dropped, and the `write_reflection`, `invoke_skill`, and `list_skills` MCP tools plus their client methods are removed. `reflections`/`skills` were consumed only by the standalone MCP memory-server, never by the Workbench runtime; reflections is superseded in spirit by the work-shaped-evals direction and skills is redundant with the `prompts` table (`017`). The capability events were a Day-1 routing-registry bet emitted by nothing — a registry schema can be re-added as a clean migration when that work is real. The `events` `event_type` enum returns to its seven live values. The Rust core is kept consistent with the schema: `core/src/events.rs` and `main.rs` drop the five capability fields from the `Event` struct and its read/write queries, the `capability_round_trip` integration test is removed, and the sqlx query cache (`core/.sqlx/`) is regenerated.

### Fixed

- Large JSON-RPC responses no longer truncate on the wire. `Deno.Conn.write` may write fewer bytes than requested, so a response larger than a single socket write was cut short and the peer's frame decoder waited forever for the missing newline (surfaced as a `sessions/list` hang). The peer now loops writes until the whole frame is flushed, with a 300 KB regression test.
- A slow JSON-RPC handler no longer blocks other requests on the same connection. The read loop awaited each handler before reading the next frame, so a long-running request wedged everything behind it (including its own cancel); handlers now dispatch concurrently while writes stay serialized, with a regression test.
- Hosted OpenAI tool calls no longer rejected for dotted command ids. The OpenAI-compatible adapter sent registry tool names verbatim, so a dotted id like `memory.read` failed OpenAI's `^[a-zA-Z0-9_-]+$` name validation with HTTP 400 — the local MLX path is lenient, so this only surfaced when routing tool turns to hosted GPT. Generalized the Anthropic wire-name sanitizer to `toolWireNames`, applied it in `buildOpenAIChatRequest` (`memory.read` → `memory_read`), and mapped the response tool-call names back to the registry name in `executeOpenAICompatibleTurn`. Round-trip tested.
- Agent loop now fires on the local model when tool calls leak as text, and the companion is grounded in its workspace. Qwen3-Coder via `mlx_lm` frequently emits its native `<function=…><parameter=…>…</function>` XML tool dialect into the message content instead of structured `tool_calls`, so the loop never fired. The OpenAI-compatible path now recovers those calls (`extractTextToolCalls`) when no structured calls are present, strips the markup, and suppresses it from the live stream, so the loop executes them — server-agnostic, per the portable-sidecar thesis. Separately, the turn-mode companion was never told its workspace root, so it guessed a stale absolute path from the loaded personal corpus; when file tools are registered the system prompt now tells the model the tools are root-scoped and paths are relative to the workspace root (without naming the absolute host path), steering it to `list_files .` instead of guessing. Verified live: the prompt that previously leaked now executes its tools and answers from the repo's `schema/`.
- `dyfj` REPL exits cleanly on Ctrl-D (EOF). Readline emits `close` on EOF but never settles the pending `question` promise, so the REPL's `await` hung and Deno reported a never-resolved top-level await instead of exiting; `readLineOrNull` now races the question against `close` and resolves null, breaking the loop. `/exit` was unaffected.
- Three latent strict-mode type errors: command policy "ask" outcomes now default a reason instead of passing `undefined` into a required field; the streaming chat reader declares the (always-absent) `toolCalls` field so both reader paths share one shape; JSON-schema tool types became type aliases so they satisfy `Record<string, unknown>` tool-parameter contracts; structured-output validation now narrows types instead of asserting past the checker.

### Changed

- `AS OF` reads resolve against Dolt commit history: events newer than the latest Dolt commit are not yet visible to time-travel queries. A periodic event-commit cadence is future work for the inspector.

# Changelog

Notable changes to DYFJ. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

DYFJ is an actively developed prototype with no release tags yet, so entries are dated rather than versioned. Document-level revisions of the operating-context README are tracked separately in its Revision history section.

## [Unreleased]

### Changed

- Local by default: a bare turn (no explicit model, tier, or hint) now always routes to the registry's local tier-0 default, even when the standing companion default names a hosted model. Hosted inference is a deliberate per-task escalation ŌĆö explicit `--model`/`/model` or `--tier`, plus the existing per-turn paid-consent gate ŌĆö never an ambient default inherited from standing config. A bare turn that bypassed a hosted configured default carries the distinct route reason `default_local` on its receipt, and a hosted turn without consent still fails closed exactly as before; a configured default that resolves to a local model is honored unchanged (`default_config`).

### Added

- Operator input is visually distinct from model output in the `dyfj` REPL. On a color terminal the entry prompt becomes a bold green `dyfj ŌØ»` gutter ŌĆö a hue the streaming markdown renderer's output palette doesn't use (bright-cyan headers, cyan code, dim receipts/markers) ŌĆö so in scrollback the operator's lines stand out as the ones carrying the green gutter, bounded below by the existing dim receipt line. With `NO_COLOR` or a non-TTY the prompt stays byte-identical to the previous `dyfj> `, so piped/scripted behavior is unchanged.
- Turn-in-flight indicator in the `dyfj` CLI. From the moment a REPL or one-shot `exec` turn is submitted until its first output ŌĆö the first streamed delta, tool/runtime status line, or mid-turn approval prompt ŌĆö a braille spinner (`ŌĀŗ workingŌĆ”`, dimmed when color is on) animates on stderr, rewritten in place with carriage-return + erase-line only; no alternate screen or raw-mode machinery. It is erased on first output, turn failure, declined approvals, and buffered-only turns, so it isn't left as an orphaned line on those paths. The spinner exists only when stderr is an interactive terminal: piped stderr and `--json` turns see no control bytes, and `NO_COLOR` drops the dim styling while keeping the animation. Lives in `prototype/src/busy-spinner.ts`.

- Context compression, so long sessions degrade gracefully instead of overflowing the context window. When a companion turn's seeded transcript would cross ~50% of the active model's context window, the elder conversation turns are replaced with a model-generated six-section summary (Session intent, Decisions & outcomes, Open threads, Key facts & references, Tool activity, Operator's words) before the first provider call; the same compressor also backs context-overflow recovery, presented through the superseding-retry stream contract. The most recent turns are always kept verbatim, and only conversation turns are ever compressed ŌĆö never the system prompt or injected memory context. Each compression records a durable `context_compressed` event (schema migration), so the summary survives resume and rebuilds the transcript the live turn used, including when the history is longer than the recent-turns cap. (That reconstruction depends on replaying the events in write order; the store orders them by timestamp rather than a causal sequence, which is stable in practice but is not enforced between identical timestamps.) Compression also appears as a visible context source in the receipt and inspector plus a runtime event on the live stream, so it is never invisible context surgery.

  **Generating the summary is always local.** The compression call routes only to an on-machine model (`ollama`/`mlx-lm`) over a direct loopback connection, chosen by locality rather than by pricing tier; it never initiates a hosted-provider request, never spends on paid inference, and declines rather than escalating when no local model is routable. The request refuses HTTP redirects, so a loopback endpoint cannot bounce the transcript body off-box. (The enforceable boundary is that direct, non-redirected loopback connection; a local inference daemon the operator runs is trusted not to relay the request elsewhere itself.)

  **Where the summary then goes:** it re-enters the conversation and is sent to the *active session model* like the turns it replaced ŌĆö which may be a hosted model. Ordinarily that discloses nothing new, since those verbatim turns already went to that same model, with two caveats worth stating plainly: the summary is pinned past the recent-turns cap, so its gist outlives the turns it replaced; and switching the session model mid-session (`/model`) can carry that gist to a provider which never saw the originals.

  The summary re-enters as one user-role message carrying a marker that names it an untrusted machine summary, produced by a compressor prompt that treats the transcript as data to summarize rather than instructions to obey. That prompt also directs the compressor to keep operator wording as attributed quotes rather than restate it ŌĆö a directive, not a validated guarantee; "Operator's words" is model-generated like the rest. A declined or failed compression never corrupts the transcript: the proactive path **continues on the uncompressed transcript**, and the reactive path leaves turn state untouched so the overflowing turn ends with the existing structured context-window-overflow error. Persistence is resolved rather than assumed ŌĆö the event write is an autocommit insert, so a rejected write is probed for durability rather than presumed lost, and the turn **fails only when durability cannot be determined at all**.

- A superseding-retry signal on the streaming turn contract, so a streaming client knows when an in-flight reply has been *replaced* rather than continued. When a retry supersedes the answer already on the wire (today: context-overflow recovery), the runtime emits `supersedingRetryStarted` between the stale and replacement deltas; the wire ordering is pinned on both stream producers and carried identically across HTTP/SSE and the UDS/JSON-RPC seam, so honoring the contract once covers both transports. Continuation retries (output-cap) deliberately never emit it ŌĆö their retry text extends the partial rather than replacing it. The `dyfj` CLI honors the signal by resetting the streaming markdown renderer ŌĆö dropping buffered-but-unrendered input and re-arming fenced-code toggling ŌĆö so the replacement answer never inherits a half-open code fence from the superseded attempt and renders as clean markdown. A dim marker makes the reset visible; lines already written to the terminal are deliberately left in place rather than cleared, because output may have scrolled beyond reach and a partial erase would misrepresent what was said. Delivery of the signal is fail-closed, unlike the best-effort status events: it is the one event a consumer must *act* on, so if it cannot be delivered to a configured event channel the superseding retry is abandoned and the turn fails, rather than streaming a replacement that would concatenate onto the stale text the consumer still has on screen. Consumers reading opaque event records get a discriminator guard (`isSupersedingRetryStarted`) that takes `unknown` and rejects malformed event payloads: each client decodes the transport JSON but does not schema-validate the decoded frame, so a `null` or primitive payload must not reach a property read. The guard checks `reason` as a string rather than against its current single value, because it is an extensible union and pinning it would make a future producer's signal silently fail to reset the render.
- Boot-time secret-pointer resolution, so `dyfj start` alone self-credentials hosted providers. A `[secrets]` config section names a vendor-neutral resolver `command` (e.g. `["op", "read"]`) and `[secrets.pointers]` maps declared secret env vars to pointers, so `config.toml` holds pointers, not values; at boot the engine runs the command per pointer and sets the value into the process environment where the providers and memory recall already read it. Resolution is session-first ŌĆö the first pending pointer, in stable declaration order, is resolved alone to warm the resolver's auth session; only a successful probe lets the rest resolve concurrently, and any probe failure (timeout, declined unlock, or a bad first pointer) skips the remainder without spawning, bounded by a single timeout. Blast radius follows that order: a non-probe pointer's failure degrades only its own provider, while a session-probe failure additionally skips the remaining unresolved pointers for that boot ŌĆö either way fail-closed with a clear, presence-only message; local-first inference is unaffected. Logging is presence-only and path-free (never a value, captured output, or the resolver/config path), and config load/validation errors report a stable label rather than the absolute path. The resolver runs in an ISOLATED environment ŌĆö spawned with a cleared env, it receives only a minimal non-secret base (`PATH`, `HOME`, `USER`, `XDG_RUNTIME_DIR`), the `[secrets.env]` plaintext map, and the names listed in `[secrets].inherit_env` ŌĆö never the runtime's other provider keys, database password, or memory token, so a compromised or misconfigured resolver's blast radius is bounded. A launch-scope resolver secret (e.g. a service-account token for unattended surfaces) is forwarded by name via `inherit_env`; `PATH`/`HOME`/linker and declared-secret names are rejected in both `[secrets.env]` and `inherit_env`. `config.toml` is documented as an executable-trust boundary: the resolver command is granted `--allow-run` and run at boot. `dyfj start` also fails closed on the trusted prototype root ŌĆö derived from `DYFJ_PROTOTYPE_ROOT` or the CLI module's own on-disk location instead of trusting the current working directory for the child runtime's permission grants ŌĆö and appends the configured resolver binary to the launch-resolved `--allow-run`, mirroring the `unix:<socket>` and memory-host net grants.
- External memory recall works against custom-header backends and private endpoints. `DYFJ_MEMORY_MCP_TOKEN_HEADER` names the header that carries the recall token: unset keeps the standard `Authorization: Bearer <token>`; set sends the raw token under the named header ŌĆö a backend that authenticates with a custom header is a config change, not a code change. The header name is declared in CONFIG_SCHEMA as ordinary config (the token itself remains a secret pointer) and granted in all three engine permission profiles, which the schemaŌćäprofile parity test enforces. `dyfj start` now derives the endpoint's `host:port` from `DYFJ_MEMORY_MCP_URL` ŌĆö resolved with the same precedence the spawned runtime uses (ambient env first, then the runtime's `.env`) so the grant cannot diverge from the URL the runtime actually dials ŌĆö and appends it to the launch-resolved `--allow-net`, mirroring the `unix:<socket>` grant, so an operator-private hostname never enters the committed permission profiles. The URL joins the CLI env allowlists (profile, compiled binary, launcher script) with a lockstep test. A malformed endpoint URL now fails `dyfj start` with a named error instead of surfacing as `NotCapable` deep inside a recall turn.
- Unpriced models are unroutable. Every spend control ŌĆö pre-call estimates, recorded actuals, the envelopes, and the runaway-anomaly hard stop ŌĆö derives from the catalog's `cost_input`/`cost_output`, so a paid-tier model whose row carries the schema-default zero cost would spend invisibly. Model routing now enforces the cost-posture rule that a model is not routable without a catalog pricing row: explicit selection or a configured default naming an unpriced paid model fails with a named error that says which row to fix, tier routing skips unpriced candidates (and says so when a tier has only unpriced ones), and malformed catalog costs parse to the unpriced bucket instead of NaN (which would fail every budget comparison open). Tier 0 is untouched ŌĆö zero is a real price for local models. `models/list` rows carry a server-computed `routable` flag and `dyfj models` annotates unpriced rows with `[unpriced ŌĆö not routable]`.
- Runaway-anomaly hard stop for paid spend. The budget envelopes are soft by design (estimate-based, confirmable, and a confirmed session/daily overrun covers its scope period); the anomaly gate closes the two blind spots that leaves, checking ACTUAL recorded spend (never pre-call estimates) before every paid call: a turn whose accumulated real cost exceeds `[anomaly].turn_multiple` ├Ś the per-call limit halts mid-turn (the multi-call agent loop can pile up real spend while each call's input-only estimate stays under the per-call check), and session/daily spend past `[anomaly].scope_multiple` ├Ś the envelope halts even after a confirmed envelope overrun (one small confirmation no longer licenses unbounded further spend for the period). Defaults 3├Ś and 2├Ś (`DYFJ_ANOMALY_TURN_MULTIPLE` / `DYFJ_ANOMALY_SCOPE_MULTIPLE`); the multiples are config-only with no per-request field, and the dollar thresholds they produce scale with the budget envelope in force ŌĆö including an explicit loopback per-turn budget override ŌĆö so the gate always binds at multiple ├Ś the effective limit. Halt semantics are deliberately harder than the ceiling gate: an approval admits the next call only ŌĆö nothing is raised, every anomalous increment re-prompts fresh, and non-interactive transports fail closed. The halt and its approval carry distinct audit bases (`policy:halt:runaway-anomaly` / `policy:allow:operator-confirmed-anomaly`). Statistical trailing-pattern detection is deferred until the receipt corpus can support it.
- Daily budget envelope. Budgets gain a third scope: total spend across ALL sessions in a local day, checked against a fresh events-table rollup before every paid call (concurrent sessions see each other's completed spend; simultaneous in-flight calls can overshoot by at most their summed cost, visible afterward in receipts) (`[budget].daily_limit_usd` / `DYFJ_BUDGET_DAILY_USD`, default $25). The same rollup also seeds the session envelope with the session's earlier turns, fixing a gap where a fresh per-turn tracker silently reset session spend to zero every turn. Ceiling confirmations cover their scope for the scope's period instead of re-prompting: a confirmed session or daily overrun holds for the rest of that session or local day ŌĆö crossing an envelope soft-confirms once, after which spend in that scope is bounded only by the per-call check, per-turn paid consent, and the agent-loop step cap until the period ends. Per-call stays a per-event check where a bigger single call prompts fresh. Confirmations live in runtime-process memory, so a restart forgets them ŌĆö the safe direction. `dyfj status` shows the daily envelope in the budget line, and per-turn budget overrides accept `dailyLimitUsd` (loopback only, same fat-finger bounds).
- `dyfj status` and `dyfj start` lifecycle commands for the local Workbench runtime. `status` reports socket reachability plus runtime/model/permission/budget posture; `start` foregrounds the Unix-socket runtime so the operator does not need to remember the underlying Deno task.
- Inter-agent coordination Slice 0 primitives: active session-coordination claims, launch packets, exit receipts, heartbeats, stale-base detection, deterministic path-overlap detection, branch-drift reconciliation, and hook-check helpers for future Workbench-native delegation. This is visibility-first coordination for concurrent agent work and gives Workbench a small substrate for making delegated work observable.
- JSON-RPC/UDS read surface now includes runtime status with grouped method catalog metadata, surface snapshots, and command/tool catalog inspection so CLI/TUI/GUI clients can share one local Workbench status substrate.
- Line-buffered streaming markdown rendering in the `dyfj` CLI output path. Companion turn text is accumulated until each newline, then styled for the terminal (headers, bold, italic, lists, inline code, fenced code blocks) with word-wrapping at terminal width and `NO_COLOR` honored. Streaming stays line-granular; `--json`, receipt lines, and approval prompts are unchanged. Renderer lives in `prototype/src/streaming-markdown.ts` with regression tests for common shapes.
- Hosted multi-model daily-driver ergonomics: a standing **paid-inference posture** (`approvePaidDefault` / `[paid].approve_paid_default` / `DYFJ_APPROVE_PAID_DEFAULT`, default off) and a **budget-ceiling warn-then-confirm** path. When a loopback turn omits `approvePaidInference`, the engine falls back to the configured default; explicit per-turn opt-in/out (`--approve-paid`) still wins, and non-loopback transports never inherit the standing posture. Budget session/per-call ceilings (`defaultSessionBudgetUsd` / `defaultPerCallBudgetUsd`) gain a config-file layer (`[budget]` in `~/.dyfj/config.toml`) alongside the existing env bindings; when projected spend would cross a ceiling the runtime warns with the numbers and requires explicit confirmation before proceeding (reusing the UDS mid-turn `approval` round-trip ŌĆö the CLI prompts on stderr). Without an interactive approver it fails closed, matching the existing approval-gate posture. The HTTP and UDS servers now load engine config at boot and thread it through the shared turn core.
- In-session model switching in the `dyfj` REPL: `/model` prints the active model and available slugs; `/model <slug>` validates against `models/list` and mutates the active routing for subsequent turns without relaunching or changing the paid posture.
- Server-console canary leak test: an integration test runs a real turn through the real UDS server path with a canary private memory injected (live Dolt, plus a loopback stub speaking the OpenAI-compatible chat/completions wire as the model), captures everything the server writes to the console, and asserts the private memory's name, its content, and the turn text never appear ŌĆö only the operational `[turn]` summary line. Verified to fail when the narration path is restored, so the invariant is guarded behaviorally, not just by review.

### Fixed

- Multibyte UTF-8 no longer corrupts when a character straddles a socket read boundary on the UDS/JSON-RPC transport. The peer read loop decoded each `conn.read()` chunk as a self-contained UTF-8 sequence, so a character whose bytes split across two reads ŌĆö a lead byte at the tail of one 4096-byte read, its continuation bytes at the head of the next ŌĆö decoded to U+FFFD on both sides. That silently mangled receipt glyphs (`ŌĆö`, `ŌåÆ`, `┬Ę`) and any non-ASCII model output streamed as deltas over the socket, in either direction, since the client and server peer share this path; it was boundary-dependent, so it surfaced only intermittently and with payload size. The loop now decodes in streaming mode, so a partial character stays buffered until its continuation arrives, and flushes the decoder at end-of-stream to finalize it ŌĆö a connection that closes mid-character yields U+FFFD for the partial bytes, which, lacking a frame terminator, is discarded with the incomplete frame. The HTTP/SSE path already streamed and was unaffected.
- A malformed tool call now comes back to the model as feedback it can act on. An invalid-arguments denial previously returned only the bare validation verdict (e.g. `missing required argument: path`) as the entire tool result, which gives the model nothing to correct against and invites a verbatim retry. The denial reason now names the tool, states the expected argument shape rendered from the tool's own schema (types, required markers, descriptions), summarizes what was received, and instructs a corrected retry. That summary names only keys declared in the schema and reports any others as a bare count ŌĆö argument values and unrecognized property names are both untrusted model input, and this reason string is persisted to the durable event log and replayed to the provider, so nothing model-controlled is echoed verbatim (CWE-532). Failed tool results are also marked as errors on the Anthropic wire (`tool_result` `is_error: true`) ŌĆö threaded through live transcripts and session-resume replay alike ŌĆö so a validation failure reads as a failure to correct rather than as ordinary tool output. The OpenAI-compatible wire has no error flag; there the enriched content alone carries the signal.
- Live external-memory recall no longer dies with `ERR_MODULE_NOT_FOUND`. The lazily imported MCP SDK transport subpath needs an explicit `.js` ŌĆö the SDK's exports map is a bare `./*` ŌåÆ `./dist/esm/*` wildcard with no extension resolution, so the extensionless specifier resolved to a file that does not exist and every recall failed at call time.
- The runtime can no longer park itself on an interactive permission prompt. Server entrypoints (`serve-unix`, `workbench`, `workbench-http`, `start`, and the `dyfj start` spawn path) now run with `--no-prompt`: an access the permission profile does not grant throws `NotCapable` (fail-closed, which the defensive code paths already assume) instead of prompting on the server's TTY. Previously the first bash tool execution of a session read env keys the profile does not grant (`LOGNAME`, `SHELL`, `LANG`, `TERM`, `TMPDIR`, ŌĆ”) and a foregrounded runtime blocked on a console prompt nobody watches while every client turn hung silently.
- Provider requests can no longer hang a turn forever. Model calls carried no timeout at all, so a blackholed connection (a VPN/mesh route flap, an IPv6 address with no route) left the turn silently stuck with nothing rendered after tool approval. Every provider fetch (OpenAI-compatible local/hosted, Anthropic, Gemini) now bounds the time to response headers (30s); once headers arrive the timer is cleared, so long streaming bodies are unaffected. A stalled connection surfaces as a named error (`<provider>/<model>: no response headers within 30000ms`) instead of an indefinite hang.
- Session resume is now discoverable end to end. `dyfj sessions` prints session slugs (`workbench-<id>`), but resuming required the bare 26-character session id, which appeared nowhere; the listing also carried no timestamps, so the session you just created was indistinguishable among older ones. `--session` now accepts either the slug exactly as listed or the bare id in any case (normalized to the canonical uppercase id) and rejects anything else with a pointer to `dyfj sessions`; the listing shows when each session last moved (newest first, as the server already ordered them) and prints a resume hint on stderr; and a new `/session` REPL command shows the current session id with the exact resume command.
- `serve-unix` refuses to bind when a live runtime already answers on the socket. Binding previously cleared any existing socket file, which silently orphaned a running runtime (it kept running, holding its Dolt pool, but became unreachable) and clients landed on whichever process bound last. The stale-socket recovery path is unchanged: a socket file nothing answers on is still cleared and bound over.
- The runtime core no longer narrates turns to the server console. `runWorkbenchRuntime` carried `console.log` presentation from its CLI origin ŌĆö context loading, workspace/model/route lines, the full model response, budget tally, and the receipt including every memory-index entry name ŌĆö and the UDS/HTTP servers invoke it directly, so all of it rendered on the server terminal ŌĆö a channel that may be logged or persisted, and that private memory context does not belong on. Turn narration now goes through an injectable `log` sink on the runtime input, default silent: the in-process CLI/shell injects console output and behaves exactly as before; transport servers stay quiet and instead emit one operational stderr line per completed turn (`[turn] session=ŌĆ” model=ŌĆ” tokens=ŌĆ” cost=ŌĆ” paid|local`) carrying routing and cost facts only ŌĆö never turn content or context-source names.
- Forward migration `schema/migrations/001_sessions_lifecycle_reconcile.sql` for databases created before the 2026-06-30 baseline refactor, which added `sessions.status`, dropped `sessions.phase`, and removed the vestigial `reflections`/`skills` tables without shipping an upgrade path ŌĆö leaving existing databases failing every session read/write with `Unknown column 'status' in 'sessions'`. The schema validator now validates two sequences: the fresh-install path (`current` + `catalog`) and the upgrade path (`history` replay + `migrations`), so forward migrations are written against the pre-baseline shape rather than needing to be no-ops against a baseline that already contains them (Dolt has no column-level `IF EXISTS` guards). Migrations README updated to match.
- `dyfj start` can actually bind the runtime socket. It previously spawned `deno task serve-unix` bare, whose permission profile cannot carry the machine-specific `unix:<socket>` net grant ŌĆö interactively Deno prompts for it, but under `dyfj start` stdin is in raw mode, so the child could not prompt and died with `NotCapable`. The start path now spawns `deno run -P=serve-unix` with an explicit `--allow-net` that reproduces the profile's declared net list (read from `deno.json` at runtime, so the two cannot drift) plus the one resolved socket path; `-P` still supplies every other permission category. A lockstep test asserts the spawn shape matches the `serve-unix` task definition.
- Test suite runs green under Deno 2.9 again: the `test` permission profile now grants `net: true`. Deno 2.9 requires an exact-match `unix:<absolute path>` net grant per socket (no directory prefixes), and the UDS integration tests bind sockets at randomized per-run temp paths, so no static allowlist can cover them and `deno.json` cannot carry machine-specific absolute paths. The relaxation is confined to the dev-only test profile; all runtime profiles keep their scoped allowlists and the structural permission-parity checks are unaffected.
- `deno task cli` now routes through the same launcher as the compiled `dyfj` client, so default Unix-socket reads such as `deno task cli models` and `deno task cli sessions` receive the exact `unix:<socket>` Deno net grant instead of falling into an interactive permission prompt.
- CLI streaming markdown renderer: underscore emphasis now requires word-boundary delimiters (CommonMark-style) so snake_case identifiers are not corrupted; header ANSI styling is re-applied after inline code spans so the rest of the line keeps header formatting.
- Large JSON-RPC responses no longer truncate on the wire. `Deno.Conn.write` may write fewer bytes than requested, so a response larger than a single socket write was cut short and the peer's frame decoder waited forever for the missing newline (surfaced as a `sessions/list` hang). The peer now loops writes until the whole frame is flushed, with a 300 KB regression test.
- A slow JSON-RPC handler no longer blocks other requests on the same connection. The read loop awaited each handler before reading the next frame, so a long-running request wedged everything behind it (including its own cancel); handlers now dispatch concurrently while writes stay serialized, with a regression test.
- Hosted OpenAI tool calls no longer rejected for dotted command ids. The OpenAI-compatible adapter sent registry tool names verbatim, so a dotted id like `memory.read` failed OpenAI's `^[a-zA-Z0-9_-]+$` name validation with HTTP 400 ŌĆö the local MLX path is lenient, so this only surfaced when routing tool turns to hosted GPT. Generalized the Anthropic wire-name sanitizer to `toolWireNames`, applied it in `buildOpenAIChatRequest` (`memory.read` ŌåÆ `memory_read`), and mapped the response tool-call names back to the registry name in `executeOpenAICompatibleTurn`. Round-trip tested.
- Agent loop now fires on the local model when tool calls leak as text, and the companion is grounded in its workspace. Qwen3-Coder via `mlx_lm` frequently emits its native `<function=ŌĆ”><parameter=ŌĆ”>ŌĆ”</function>` XML tool dialect into the message content instead of structured `tool_calls`, so the loop never fired. The OpenAI-compatible path now recovers those calls (`extractTextToolCalls`) when no structured calls are present, strips the markup, and suppresses it from the live stream, so the loop executes them ŌĆö server-agnostic, per the portable-sidecar thesis. Separately, the turn-mode companion was never told its workspace root, so it guessed a stale absolute path from the loaded personal corpus; when file tools are registered the system prompt now tells the model the tools are root-scoped and paths are relative to the workspace root (without naming the absolute host path), steering it to `list_files .` instead of guessing. Verified live: the prompt that previously leaked now executes its tools and answers from the repo's `schema/`.
- `dyfj` REPL exits cleanly on Ctrl-D (EOF). Readline emits `close` on EOF but never settles the pending `question` promise, so the REPL's `await` hung and Deno reported a never-resolved top-level await instead of exiting; `readLineOrNull` now races the question against `close` and resolves null, breaking the loop. `/exit` was unaffected.
- Three latent strict-mode type errors: command policy "ask" outcomes now default a reason instead of passing `undefined` into a required field; the streaming chat reader declares the (always-absent) `toolCalls` field so both reader paths share one shape; JSON-schema tool types became type aliases so they satisfy `Record<string, unknown>` tool-parameter contracts; structured-output validation now narrows types instead of asserting past the checker.

### Changed

- Schema files are now split into a readable current baseline (`schema/current/`), mutable catalog seeds (`schema/catalog/`), forward migrations (`schema/migrations/`), and preserved replay history (`schema/history/`). `deno task validate-schema` applies the current path and separately replays history, keeping the Dolt audit trail without forcing readers through every historical migration.
- Compiled `dyfj` CLI net grants for Deno 2.9 dynamic UDS paths. Deno 2.9 requires an exact-match `unix:` net grant per socket path; the compiled binary still bakes the default `~/.dyfj/run/workbench.sock` grant at compile time for speed, but `deno task compile-cli` now ships a thin launcher (`dist/dyfj`) that execs the compiled `dist/dyfj-bin` on the default path and falls back to `deno run` with a runtime-resolved `unix:<sock>` grant when `DYFJ_SOCKET` or `XDG_RUNTIME_DIR` shifts the path (mirroring the serve-unix bind-side pattern). Net permission stays scoped to loopback HTTP plus the one UDS path in use; no literal host path is committed. Regression coverage lives in `prototype/scripts/dyfj-launcher.test.ts`.
- Budget enforcement at the spend boundary is warn-then-confirm rather than an immediate hard deny when an approver is available; non-interactive transports still fail closed at the ceiling.
- Budget-ceiling confirmation is deduped to once per turn for the same projected spend level (preflight + per-call gate no longer double-prompt); a later agent-loop call re-prompts only when projected spend rises above what was already confirmed.
- Budget-ceiling dedupe tracks per-call and session projections independently so a later call that crosses the session limit re-prompts even when per-call was already confirmed at the same estimate.

- A declared configuration key registry (`CONFIG_SCHEMA` in `prototype/src/config.ts`) ŌĆö the single source of truth for the system's env surface. Each key declares its env-var binding, its **engine vs client domain** (the engine owns runtime/data/secret config; the engine-free CLI owns its transport/routing slice), its type, and whether it is a **secret pointer** (a key carrying a credential ŌĆö an `op://` ref / keychain item / env-var name ŌĆö resolved at point of use, never stored as a plaintext value on the config surface). `DYFJ_PRINCIPAL_ID` is declared but flagged `sessionState`: per the configuration-system thesis the principal rides the connection (the UDS peer's OS identity locally, the tailnet identity remotely), so it is _not_ config ŌĆö it is declared only so the permission allowlist can account for it. Two consumers move onto the declared surface in this slice (below); migrating the remaining env-read sites continues incrementally.
- The permission-parity check became **structural**. The hand-maintained `deno.json` permission `env` arrays are now asserted against the declared `CONFIG_SCHEMA` surface (forward: every declared engine env var is granted by each turn-running engine profile ŌĆö `workbench`, `workbench-http`, `serve-unix` ŌĆö so no profile can silently lag the runtime, the live failure that surfaced `DYFJ_PRINCIPAL_ID` over the socket; reverse: no profile may grant an undeclared runtime env var, so a new var can't reach the allowlist without joining the declared surface). This replaces the band-aid `serve-unix ŌŖć workbench-http` env parity test (moved from `uds-server.test.ts` to `config.test.ts`); the `serve-unix ŌŖć workbench-http` **net** backstop is retained until net-host derivation from the schema lands.
- Budget defaults migrated onto the declared config surface. `budget.ts` no longer inlines the `DYFJ_BUDGET_SESSION_USD` / `DYFJ_BUDGET_PER_CALL_USD` names and the `$1.00` / `$0.10` literals ŌĆö those live in `CONFIG_SCHEMA` and resolve through `resolveBudgetDefaultsFromEnv` (defaults ŌåÆ env, rejecting non-numeric or negative limits rather than silently coercing to `NaN`). They are resolved **once at the runtime boundary** (`resolveRuntimeEnvDefaults`, the same place the principal, root, and budget-tally mode resolve) and threaded into the turn, so the core reads no env for budget; precedence is per-turn override ŌåÆ boundary-resolved default ŌåÆ the declared `BUDGET_DEFAULTS`. `DYFJ_PRINCIPAL_ID` resolution is likewise centralized behind one `resolvePrincipalId` resolver (the boundary plus the deep event-attribution fallback), retiring the scattered `process.env.DYFJ_PRINCIPAL_ID ?? process.env.USER` reads. The budget config-_file_ layer and wiring `loadConfig` into the CLI/HTTP entrypoints (today only `serve-unix` loads the file) are the next config slice.

- An `operator` permission profile that auto-approves contained mutating tools instead of prompting per-call (`permissionLevel` config, default `strict`). The mid-turn approval gate is fail-closed but unusable for daily companion use ŌĆö every `write_file` (and soon `edit_file`/`bash`) prompts. The config knob now drives the command policy: on a **loopback/operator turn** with `permissionLevel = "operator"`, a _contained_ mutation (local, free, workspace-write) is auto-approved with its own audit basis (`policy:allow:operator-profile`); paid, networked, or command-execution tools are deliberately **not** covered and still prompt, so a future `bash` stays gated until its own safety model says otherwise. `strict` (the default) keeps the per-call approval gate exactly as-is. The level resolves once at the boundary and threads to `evaluateCommandPolicy` alongside the loopback transport ŌĆö a non-loopback consumer never receives the relaxed posture. Verified end-to-end: under `operator` a non-interactive write is auto-approved and written; under `strict` the same write is denied.

- Typed engine configuration with a TOML config file (`~/.dyfj/config.toml`). The runtime's startup posture ŌĆö the defaults it uses when a request doesn't specify ŌĆö now resolves from a single declared surface with precedence **defaults ŌåÆ config file ŌåÆ environment**, replacing scattered per-consumer env reads for the first knobs. The file is TOML: hand-edited, comments, idiomatic for the future Rust core. Config holds startup posture, never session state; anything the app writes (machine state) lives elsewhere, so the hand-edited file stays pristine and comment-preserving writes are a non-problem. First two knobs: `defaultCompanionModel` (the model used on a "bare turn" with no model/tier/hint ŌĆö set a frontier slug for the companion; threaded through `selectWorkbenchModel` ahead of the registry's local default) and `permissionLevel` (`strict` | `operator`, default `strict`; the policy honoring `operator` lands with the permission-profile work). Loaded **once at the engine boundary** (the UDS server) and passed into the runtime ŌĆö no hidden per-turn reads; a malformed config fails the boot loudly rather than silently mis-configuring. The TOML parser is imported lazily so the module loads cleanly under the test runner. Migrating the remaining env-read sites and deriving the `deno.json` permission allowlist from the schema are follow-on slices.

- `search_memory` ŌĆö vendor-neutral recall over an external long-term memory. The turn path can now query an operator-configured external memory by meaning (semantic search) on demand, complementing the curated context it injects. DYFJ owns the capability (the `search_memory` tool); the backend is a config value, never a vendor named in the code ŌĆö any memory that speaks MCP and exposes a search tool is reachable by setting `DYFJ_MEMORY_MCP_URL` (with `DYFJ_MEMORY_MCP_TOOL`, default `search`, and an optional `DYFJ_MEMORY_MCP_TOKEN`), so swapping the backing store is a config change (or, for a non-MCP backend, one adapter behind `MemorySearch`), never a rename through the codebase. Protocol-as-firewall applied to memory: bet on the protocol, keep vendor optionality. The tool is read-only, opt-in (an unset URL means it is never offered), and offered only on a loopback/operator turn ŌĆö a non-loopback consumer never receives it, so the external memory is unreachable off-box. Its egress gets a dedicated `recall` policy tier (`policy:allow:operator-configured-recall`): auto-allowed without a per-call approval prompt, because the destination is operator-configured and fixed (the model chooses the query, not where it goes) and the tool is registered only for permitted turns. The MCP SDK is imported lazily so the capability adds no load-time cost and the module loads cleanly under the test runner. Enabling recall requires granting the runtime network access to your endpoint.

- Memory injection is now scoped by a curated **inject classification**, not by type (`schema/024_memories_inject.sql`). The turn-mode path previously bulk-injected the full content of every `user`+`feedback` memory on every turn (~63KB / tens of thousands of tokens), filtered only by privacy clearance ŌĆö with no relevance or recency scoping. That corpus was a frozen one-time seed with no live source, so it injected stale material (defunct vocabulary, ephemeral dated session-state checkpoints) as if current: staleness as a correctness failure, not just token cost. The `memories` table gains an `inject` column (`always` | `index` | `never`, default `index`): `always` is the small curated worldview (identity core + operating preferences) injected as content; `index` is slug+name+description only, pulled on demand via `read_memory`; `never` is withheld from both. The runtime loads `inject='always'` as the core context and `inject='index'` as the pull-on-demand index (`loadInjectedMemories` / `loadIndexedMemories`), each still clearance-scoped ŌĆö replacing the former type-based bulk load. Injection drops from the full personal pile to a tight, deliberately-curated set; everything else stays reachable without being dumped. The durable source fix ŌĆö regenerating the curated rows from a maintained context source of truth instead of a frozen seed ŌĆö is the next slice.
- The `dyfj` CLI can approve mutating tools over the `--unix` seam. When a turn over the socket hits a tool that needs approval (e.g. `write_file`), the runtime's mid-turn `approval` request is answered by an interactive `y/N` prompt that shows the tool, its arguments, and a content preview ŌĆö printed to stderr so a `--json` turn's stdout stays clean. Non-interactive runs (no TTY) deny without prompting, fail-closed. `connectUnixClient` gains an `onApproval` hook that registers the peer's `approval` request handler; with no approver the server's request fails and it denies. This makes an approved `write_file` over `dyfj --unix` a live, human-in-the-loop operation.
- Mutating tools behind a mid-turn approval gate ŌĆö `write_file` + the approval round-trip. The command policy already routes write-filesystem to `ask`; the runtime now resolves `ask` by calling an injected `confirmToolApproval(request) ŌåÆ approve | deny`, executing only on approval (authz `policy:allow:operator-approved`) and otherwise denying (`policy:deny:approval-denied`). The default is **deny** ŌĆö fail-closed, like the paid-escalation gate ŌĆö so a missing approver never runs a mutation, and a malformed call is rejected before approval is even requested. Over the UDS/JSON-RPC seam the approver is a server-initiated `ctx.request("approval", {tool, callId, arguments, title})` answered by the connected client (the mid-turn duplex round-trip the seam was built for); HTTP/SSE has no such channel, so mutating tools deny there. The first mutating tool, `write_file`, writes UTF-8 within the workspace root (symlink-safe; parent must exist), registered alongside the read-only file tools. Payload-bearing arguments are redacted from the persisted tool-call event and session replay ŌĆö a `redact` schema flag replaces the value with a constant `[redacted]` sentinel (any runtime type, including malformed or denied calls), so `write_file` content never reaches the durable log or replay (CWE-532) ŌĆö centralized so future mutating tools inherit it. `bash` + `edit_file` ride this gate in a later slice.
- Transport seam ŌĆö interactive `turn` over the JSON-RPC/UDS duplex channel. The security-sensitive turn path (request resolution, per-session serialization, resume reconstruction, paid-escalation gating, runtime invocation) was lifted out of the HTTP driver into a transport-neutral `turn-runner.ts`, so the HTTP/SSE veneer and the UDS/JSON-RPC seam run the identical turn through one shared core instead of duplicating the money/audit/clearance logic per transport ŌĆö a pure refactor proven faithful by the existing HTTP suite (the transport-parity and per-session-lock TOCTOU tests stayed green). The UDS server gains the `turn` method on that core: intermediate text deltas and runtime events stream back as `stream` notifications whose params reuse the HTTP SSE frame shape, so a client shares one frame handler across transports, and the receipt is the RPC result. UDS is the canonical `loopback` transport, so paid inference and per-turn budget overrides remain available ŌĆö but only with the explicit per-turn opt-in the shared core enforces (loopback + opt-in to spend; denied otherwise) ŌĆö covered by tests for paid approve/deny, budget override, streaming, and invalid params. RPC handlers also gained a per-connection context (`notify` + a server-initiated `request`) so a running handler can both stream and issue the mid-turn approval request ŌĆö the mechanism the approval keystone builds on.
- Transport seam ŌĆö JSON-RPC 2.0 core and Unix-domain-socket transport. A hand-rolled, SDK-free JSON-RPC 2.0 core (`jsonrpc.ts`: newline-framed wire, message classification, dispatch, and the DYFJ fail-closed application error codes) and a bidirectional peer (`jsonrpc-peer.ts`) over any byte-stream connection, with a UDS server (`uds-server.ts`) serving the read methods (`models/list`, `sessions/list`, `events/query`) through the same runtime functions the REST endpoints use. The engine-free `dyfj` CLI reaches those methods over the socket (`--socket` / `DYFJ_SOCKET`), and a `serve-unix` launcher (plus root-level `serve-unix` / `cli` task proxies) serves the seam; the socket path resolves `DYFJ_SOCKET` ŌåÆ `$XDG_RUNTIME_DIR/dyfj` ŌåÆ `~/.dyfj/run`, with the parent directory created mode 0700 and stale-socket clearing that refuses to unlink a non-socket. UDS is the canonical `loopback` transport per the transport-seam decision (2026-06-21); a committed runnable spike (`examples/uds-jsonrpc-spike.ts`) proves the duplex round-trip.
- Live streaming through the agent loop + a thrash guard. The OpenAI-compatible streaming reader now parses tool calls out of the SSE stream (indexed `tool_call` deltas, accumulating fragmented arguments), so a streamed turn can both stream text and request tools. The runtime streams tool-offering calls on providers whose wire path supports it (`modelStreamsToolCalls` ŌĆö local MLX/Ollama + hosted OpenAI; Anthropic/Google stay buffered, since their streaming readers don't carry tool calls, and tool calls are captured from buffered JSON there). This also restores live token streaming for ordinary companion replies ŌĆö which the Slice A file-tool registration had silently disabled by suppressing streaming whenever tools are projected ŌĆö and streams each gather step live. Adds an anti-thrash guard: when the model emits a whole step of tool calls it already made this turn, the loop drops tools and forces a concluding answer instead of spending steps up to the cap. Verified live on the local 30B coder: streamed tool calls are captured and executed, normal replies stream, and a multi-hop task that previously ran to the 8-step cap now concludes in three.
- Agent loop ŌĆö multi-step tool iteration. The runtime now loops modelŌåötools until the model stops requesting tools or a step cap (`MAX_TOOL_STEPS`, 8) is reached, replacing the single tool round (`src/workbench.ts`). Tool-gathering steps run buffered (the streaming readers don't parse tool-call deltas yet ŌĆö tracked as a follow-up), with progress surfaced via a per-step log and the `tool_call`/result events; at the cap the runtime drops tools to force a concluding answer, which streams. The follow-up prompt accumulates results across steps and tells the model not to repeat calls it already made. Verified live against the local Qwen3-Coder-30B: a batched single step, genuine sequential multi-step iteration, and the cap's forced conclusion. Regression-tested with a mock provider (natural termination + cap). Budget enforcement and accounting are per provider call, not per turn: every call in the loop is budget-checked before and recorded after, so per-call and session limits hold across steps and receipts/audit events aggregate every call ŌĆö closing a pre-push finding that the loop could make several hosted calls under one preflight and record only the final one's usage.

### Security

- Recall endpoints must be https off-box, and recall requests refuse redirects. A configured `DYFJ_MEMORY_MCP_URL` using plain http is rejected unless the host is genuinely loopback (`localhost`, IPv6 loopback, or a strict dotted-quad in `127.0.0.0/8` ŌĆö a DNS name merely starting with `127.` does not qualify), enforced at both seams: config resolution (the runtime fails closed before any request is built) and the launcher's net-grant derivation (a cleartext destination is never granted). Every recall request carries `redirect: "error"`, so an accepted https endpoint answering a 307/308 downgrade fails loudly instead of being followed ŌĆö fetch preserves custom headers across redirects, and following one could ship the token header and query body in cleartext. The recall token rides a request header and the query is private data; neither crosses the network unencrypted.
